# Linux 运维 Agent 软件产品说明书

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 产品名称 | Linux 运维 Agent |
| 产品形态 | 本地 CLI、Web 控制台、GitHub Release 临时 Remote CLI/Web |
| 当前实现基线 | 2026-07-11 仓库状态 |
| 目标环境 | 具备 Bash、Python 3、jq、curl、GNU/BusyBox timeout 等基础工具的 Linux 主机 |
| 核心定位 | 保守、可审批、可审查、可审计的 Linux 运维辅助 Agent |

本文说明当前软件产品能力、运行边界、用户流程、扩展机制、配置、部署和验收标准。实现细节以仓库源码和测试为最终依据。

## 2. 产品概述

Linux 运维 Agent 将自然语言请求转换为结构化回答或执行计划，并在真正执行前经过独立的参数校验、策略审查、风险分级、人工审批、最小权限代理、运行观察和审计记录。

产品遵循以下原则：

- CLI 是核心能力入口，Web 是复用同一机器 API 的本机控制台，不复制执行逻辑。
- 模型输出是不可信数据。模型只能提出计划，不能绕过 policy、approval、observer 和 audit。
- 只读操作优先使用登记后的 Skill；文件修改、下载、网络探测和系统变更使用受控工具边界。
- 高风险、关键路径、关键服务、隐藏执行和未知能力默认要求审批或直接阻断。
- 本地与 Remote 运行共享同一策略和执行链路，Remote 不降低安全要求。

## 3. 目标用户与使用场景

### 3.1 目标用户

- Linux 系统管理员和 SRE。
- 需要标准化巡检、诊断和审计留痕的运维团队。
- 需要通过本机 Web 页面辅助操作，但不希望部署数据库或复杂服务端依赖的用户。
- 需要在临时节点上运行一次性诊断，并在退出后自动清理运行时的用户。

### 3.2 主要场景

- 磁盘、CPU、内存、进程、服务、日志、网络和文件句柄诊断。
- 受控日志检索、清理规划、备份和安全截断。
- 经授权的网络扫描、DNS、SNTP、WHOIS、SNMP、防火墙规划等操作。
- 生成、审查、人工编辑和保存新的 Skill。
- 审查 shell 命令、Skill 脚本、远程脚本和 MCP tool 调用。
- 查看审计 session、执行时间线、审批决定、observer 状态和输出摘要。
- 通过 MCP 接入外部工具目录，并将工具调用纳入统一审批链路。

## 4. 产品运行形态

| 入口 | 启动方式 | 交互特点 | 状态与数据 |
| --- | --- | --- | --- |
| 本地 CLI | `bash bin/agent` 或子命令 | REPL 或单次命令；从终端读取审批 | 使用本地 config、skills、mcp、logs、tmp |
| 本地 Web | `bash bin/agent-web` | 浏览器工作台、异步 job、审批抽屉 | Web 通过 `bin/agent api` 调用核心能力 |
| Remote CLI | Release bootstrap 管道 | 从 `/dev/tty` 读取审批和可选 API key | 核心与 Skill 优先放入内存/临时运行时，退出清理 |
| Remote Web | Release bootstrap 管道 | 强制监听 `127.0.0.1`，远程访问使用 SSH 转发 | 临时 token、临时运行时、显式备份后才能持久化 |

Remote bootstrap 只接受同一官方 Release manifest 中登记的资产，验证文件名、大小、SHA256 和归档边界。首次启动仅加载 core、策略、prompt 和 Skill 索引；完整 Skill 包按需物化。

## 5. 工作模式与用户流程

### 5.1 Work 工作模式

命令：`bash bin/agent work "<需求>"`

1. 记录 `received` 事件并识别请求主题。
2. 采集最小必要环境上下文并脱敏。
3. 附加 Agent loop 状态、渐进式披露的 Skill 指令和 MCP 工具目录。
4. 请求模型返回 `answer` 或符合 schema 的 `work_plan`。
5. 对每个步骤验证 executor、Skill/MCP 登记状态和 JSON 参数。
6. 运行 policy review；Critical 直接阻断，其余按风险和配置决定自动批准或人工审批。
7. 通过 observer 执行并记录结果；单步骤受 `execution.timeout_sec` 硬超时保护。
8. 当计划声明仍需深入时生成受限 observation，并进入下一轮反思。
9. 每隔 `checkpoint_turns` 请求用户确认；无论用户是否持续批准，都受 `max_iterations` 总轮数上限约束。

Work 支持的 executor：

- `skill_script`：登记后的本地或 Remote Skill 脚本。
- `shell`：Skill 无法覆盖时使用的 shell 命令。
- `remote_script`：先下载、校验、审查，再审批执行的 HTTPS 脚本。
- `mcp_tool`：已安装、启用并通过 `tools/list` 发现的 MCP 工具。

### 5.2 Edit 编辑模式

命令：`bash bin/agent edit "<需求>"`

1. 模型生成 `skill_edit` JSON，包括 Skill 名称、描述和脚本正文。
2. 系统对脚本名、内容和策略风险做预审。
3. CLI 打开 `$EDITOR` 或 `vi`；用户可以修改或取消。
4. 系统记录人工修改摘要，再次审查脚本和 Skill 结构。
5. 候选 Skill 必须包含参数规范，脚本必须在 `SKILL.md` 与 `INDEX.md` 中登记。
6. 校验通过后原子写入 Skill 目录；失败不会留下半成品。

Web 对应流程为 edit plan、review、apply。Web 不直接执行 MCP tool；MCP 只作为生成 Skill 时的外部能力参考。

### 5.3 Script 命令模式

命令：`bash bin/agent script <skill>/<script> '<json-object>'`

- ref 必须符合 `skill/script` 格式，并同时存在于 `skills/INDEX.md`、对应 `SKILL.md` 和 `scripts/`。
- 唯一参数必须是 JSON object；脚本 stdout 应只输出一个 JSON object。
- 系统结合脚本声明风险和策略发现项决定阻断、审批或自动执行。
- Remote 模式下，未物化的 Skill 在执行前按一级 Skill 包下载和校验。

### 5.4 Terminal 命令模式

命令：`bash bin/agent terminal "<shell 命令>"`

- 命令先经过结构化 command guard 和 JSON 风险规则。
- 自由 shell 文件修改、隐藏写入、受保护路径操作和破坏性命令被阻断。
- `command`、`builtin`、shell `-c/-lc` 和可解析 `timeout` 会展开真实子命令继续审查。
- `eval`、`source`、不透明 forwarder、`find -exec` 等可隐藏任意执行的形态直接阻断。
- 低风险 shell 默认仍需人工审批；是否自动执行由 `approvals.auto.shell_readonly` 控制。
- 执行超过 `execution.timeout_sec` 时返回 `timed_out`，先发送 TERM，5 秒后仍未退出则强制结束。

### 5.5 Doctor、Sense、Audit、Policy 与 Backup

- `doctor`：检查必需命令、配置 JSON、Skill 校验和 Remote 状态。
- `sense`：按主题采集最小必要环境信息。
- `audit <session-id>`：读取 JSONL 并生成审计报告。
- `policy validate`：验证风险、脱敏和审计边界文件。
- `backup <output.tar.gz>`：显式导出脱敏后的 Remote 运行时资产。

## 6. Web 控制台

Web 后端仅使用 Python 标准库，默认监听 `127.0.0.1:8765`。静态资源可直接读取，所有 `/api/` 请求必须携带 Bearer token 或受支持的 token header。

主要页面：

- Work：任务输入、terminal、时间线、输出区块、审批和继续执行。
- Skill：脚本目录、Markdown 预览、执行、生成、审查、保存和校验。
- MCP：server manifest、脱敏配置、连接校验和 tools catalog。
- Policy：策略读取、校验和经 sudo 审批的写入。
- Audit：session 列表、事件筛选、指标、报告和时间线恢复。
- Config：白名单配置项、Provider/模型、审批开关、observer、执行超时和 Agent 轮数上限。

长任务使用异步 job。用户可以轮询状态、查看增量 stderr 摘要和取消进程组。Web Agent session 由 Web 后端管理，CLI 子进程将审计写入该 session，不重复创建 session 起止事件。

## 7. 系统架构

```text
CLI / Web / Remote adapters
          │
          ▼
Orchestrator + AI + Context + Sense
          │
          ▼
Schema validation + Skill/MCP registry
          │
          ▼
Policy review + Approval + Least privilege
          │
          ▼
Executor + Timeout guard + Observer
          │
          ▼

```

核心模块职责：

| 模块 | 职责 |

| `lib/orchestrator.sh` | Work/Edit/Script/Terminal 编排和 Agent loop |
| `lib/executor.sh` | 步骤状态机、审批、执行、恢复和输出协议 |
| `lib/command_guard.py` | shell token/结构风险分类和嵌套命令审查 |
| `lib/policy.sh` | 风险规则、策略文件和审计边界校验 |
| `lib/observer.sh` | auditd observer 生命周期和受限进程执行 |
| `lib/audit.sh` | 脱敏 JSONL、事件摘要、报告和边界应用 |
| `lib/skills.sh` | Skill registry、渐进式披露、风险声明和校验 |

## 8. 策略、审批与阻断

### 8.1 风险结果

Policy review 输出统一包含：

- `approved`：策略是否允许进入后续流程。
- `approval_required`：是否必须人工确认。
- `risk_level`：`low`、`medium`、`high` 或 `critical`。
- `findings`：带 code、severity、category、message 的发现项。

Critical finding 使 `approved=false` 并阻断。Medium/High 或配置要求的 Low 操作进入人工审批。MCP tool 默认不能静默自动执行。

### 8.2 重点阻断边界

- 根目录和 `/etc`、`/boot`、`/usr`、`/var/lib`、`/root`、用户 `.ssh` 等保护路径。
- 删除、磁盘破坏、任意文件写入、权限/所有权修改和重定向写入。
- `curl|sh`、`wget|sh`、未经审查的远程脚本。
- 关键服务停止、禁用或破坏性操作。
- shell/解释器隐藏执行、命令替换、进程替换和不透明 forwarder。
- 未登记 Skill、不可用 MCP server/tool、非 object 参数。

### 8.3 执行资源边界

- AI 请求由 `request_timeout_sec` 限时。
- MCP 连接由 manifest `timeout_sec` 限时，范围 1–120 秒。
- 所有实际执行步骤由 `execution.timeout_sec` 限时，范围 1–3600 秒。
- Agent loop 由 `max_iterations` 限制在 1–100 轮，并保留周期 checkpoint。
- observer 和审计输出有事件数、文本长度和字段白名单限制。

## 9. 审计与 Observer

### 9.1 审计数据

每个 session 使用一份 `logs/<session-id>.jsonl`。事件覆盖：

- session、command、turn 和 control 生命周期；
- received、sensed、request context、AI 失败和计划；
- policy checked、approval、blocked、running、succeeded、failed、skipped；
- Agent loop、reflection、checkpoint 和硬上限停止原因；
- edit、script、terminal、remote、backup 和 Skill materialization；
- observer session/execution 状态和 AI 实际披露文件 manifest。

入口事件只记录命令名和参数数量，真实业务输入由 `received` 记录，避免同一输入在 session/command/received 三处重复保存。`request_context_built` 的安全摘要不重复保存请求正文，只记录会话轮数、披露 Skill、不可用 Skill、MCP server/tool 数和发现项数量。

### 9.2 审计边界

`policies/audit-boundaries.json` 同时定义：

- `observing`：当前希望观察的事件、syscall、结果字段、文本和事件上限。
- `allowed_to_observe`：允许配置触达的最大边界。

策略校验会阻断以下错误：

- payload mode 不在允许列表；
- 事件、syscall 或结果字段不是字符串数组；
- `observing` 选择了 `allowed_to_observe` 之外的值。

因此配置不会再出现“校验成功但运行时静默丢弃”的漏记状态。

### 9.3 脱敏

日志写入前根据 `redaction-rules.json` 处理私钥、Authorization、Cookie、API key、token、password、JWT、常见 token 前缀、长十六进制值和私网 IPv4。敏感 key 的 JSON value 会替换为 `[REDACTED]`。

### 9.4 Observer

observer 在允许且可用时使用 auditd 记录受限 syscall 和事件样本；不可用、权限不足或用户跳过时写入明确的降级原因。执行事件包含开始/结束时间、root pid、exit code、timeout 状态和最小权限代理信息。

## 10. Skill 能力系统

### 10.1 两阶段渐进式披露

1. 发现层：模型系统提示只附加 `skills/INDEX.md`，用于查看 Skill 名称和用途摘要。
2. 指令层：系统根据当前请求主题或明确 Skill 名称，只将命中的 `SKILL.md` 放入 `request_context.skills.disclosed`。

每次实际披露的文件路径、大小和 SHA256 都写入 `ai_files_manifest`。未命中的 Skill 不会传给模型；Remote 中未物化的 Skill 不会为了构造提示而自动下载，并在 `unavailable` 中明确显示。

### 10.2 SKILL.md 参数规范

每个 Skill 必须包含参数契约，至少明确：

- 唯一位置参数为 JSON object 字符串；
- 每个字段的类型、必填性、默认值和范围/枚举约束；
- stdout 的单 JSON object 契约和 `ok/status/error` 语义；
- 授权确认、dry-run、apply、confirm 和敏感字段边界；
- 至少一个合法调用示例。

`skills validate` 会对缺失参数契约、脚本文件、INDEX 登记、SKILL 登记和风险阻断项给出 Critical finding。

### 10.3 内置 Skill 分类

| Skill | 能力范围 |
| --- | --- |
| `ops-basic` | 基础资源、进程、服务、日志、备份和清理规划 |
| `os-deep-inspect` | OS 快照、连接、文件句柄和 journal 深度诊断 |
| `controlled-tools` | 文件匹配、原子补丁、安全下载和本地分析 |
| `session-history` | 审计 session 中上一轮/指定轮次输出回看 |
| `network-ops-tools` | 受限网络诊断、扫描、查询、规划和计算 |

风险声明从 `SKILL.md` 中“脚本名与 risk 同行”的记录解析，不依赖文档中第一次出现脚本名的位置。

## 11. MCP 能力系统

### 11.1 Manifest

路径：`mcp/<server-id>/mcp.json`

支持：

- `stdio`：子进程 stdin/stdout 的逐行 JSON-RPC。
- `streamable_http`：单 HTTP endpoint，响应支持 JSON 或 SSE。
- `sse`：兼容旧版 HTTP+SSE 双端点。

manifest 可以包含 command、args、env、URL、headers、message URL、enabled 和 timeout。API/Web 返回时对 secret-like 字段脱敏。

### 11.2 生命周期和协议

客户端执行 `initialize`、`notifications/initialized`、`tools/list` 和 `tools/call`。当前首选协议版本为 `2025-11-25`，并仅接受客户端明确支持的协商版本；未知版本会断开并返回错误。

Streamable HTTP：

- POST Accept 同时声明 `application/json` 和 `text/event-stream`。
- 保存 `MCP-Session-Id` 并在后续请求发送。
- 后续请求携带协商后的 `MCP-Protocol-Version`。
- 支持 JSON 和 SSE response。

Legacy SSE：

- GET stream 和 POST message endpoint 都携带 manifest 自定义 headers，包括 Authorization。
- endpoint event 可以提供 message URL。

实现依据 MCP 官方稳定规范：

- <https://modelcontextprotocol.io/specification/2025-11-25/basic/transports>
- <https://modelcontextprotocol.io/specification/2025-11-25/basic/lifecycle>

### 11.3 MCP 安全边界

- 外部 tool description、schema 和 annotations 只作为不可信 metadata。
- tool 必须存在于实时 `tools/list`，不能由模型臆造。
- arguments 必须是 JSON object。
- 实际调用只能作为 Work 的 `mcp_tool` step，不能从浏览器直接任意调用。
- 调用仍经过 policy、approval、timeout、observer 和 audit。

## 12. 配置说明

关键配置：

| 配置 | 默认 | 说明 |
| --- | --- | --- |
| `request_timeout_sec` | 90 | 单次 AI 请求超时 |
| `context_turns` | 6 | 会话历史窗口 |
| `agent_loop.enabled_for_work` | true | 启用 Work 反思续写 |
| `agent_loop.checkpoint_turns` | 0 | 0 表示使用上下文窗口作为 checkpoint 周期 |
| `agent_loop.max_iterations` | 12 | Agent loop 硬轮数上限，1–100 |
| `execution.timeout_sec` | 300 | 单执行步骤硬超时，1–3600 秒 |
| `execution.min_privilege_proxy` | true | root 运行时尽量降权 |
| `execution.least_privilege_user` | nobody | 降权目标用户 |
| `audit_mode` | safe_summary | 安全摘要或脱敏详细模式 |
| `audit_text_limit` | 1000 | 审计文本摘要上限 |
| `observer.enabled` | auto | 自动、强制 auditd 或禁用 |
| `observer.max_events` | 200 | observer 事件上限 |
| `remote_script_policy` | download_review | 下载审查或禁用 |
| `remote.allow_api_key_transmission` | false | Remote 是否允许向 AI Provider 发送 API key |

API key 优先从 `LINUX_AGENT_API_KEY` 读取，其次读取 `config.api_key`。Web API 不回显密钥正文。

## 14. API 与状态语义

机器 API 形式：

```bash
bash bin/agent api <resource> <action> '<json-object>'
```

常见状态：

- `answered`：Work 直接回答。
- `executed`：执行完成。
- `approval_required`：等待用户决定，返回 approval card 和 execution state。
- `blocked`：策略或登记边界阻断。
- `rejected` / `terminated` / `checkpoint_stopped`：用户决定停止。
- `iteration_limit_stopped`：达到 Agent loop 硬上限。
- `timed_out`：执行步骤达到硬超时。
- `ai_failed` / `ai_invalid_response`：模型请求或 schema 失败。

Web 和 CLI JSON 模式统一使用 timeline、approval card 和 output blocks 表达计划、审查、执行、observer 和输出。

## 15. 故障处理

| 现象 | 检查项 |
| --- | --- |
| AI 请求失败 | `api_url`、model、API key 来源、Remote 传输开关、`request_timeout_sec` |
| Skill 未找到 | INDEX、SKILL.md、scripts 文件、Remote materialization、`skills validate` |
| MCP 无工具 | manifest enabled/transport、网络/进程、headers、协议版本、`mcp tools` |
| 一直等待审批 | 查看 `approval_required`、Web approval card 或 CLI `/dev/tty` |
| 执行超时 | 调整 `execution.timeout_sec`，检查命令是否本应使用有界 Skill |
| observer 降级 | sudo/auditctl/ausearch、observer privilege、审计中的 reason_code |
| 策略配置不生效 | 运行 `policy validate`，检查 observing 是否超出 allowed_to_observe |
| Web 无法访问 | 确认仅监听 localhost、端口、Bearer token 和 SSH 转发 |

## 17. 已知边界

- command guard 是保守分类器，不是完整 Bash 解释器；无法安全解析的隐藏执行会被阻断。
- Web 面向本机使用，不应直接监听公网地址；跨主机访问应使用 SSH 转发。
- auditd observer 依赖主机工具和权限；不可用时产品只能提供应用层审计与明确降级事件。
- Legacy SSE 仅用于兼容旧 MCP server，新部署应优先使用 Streamable HTTP。
- Agent 不代替运维人员的授权判断、变更窗口、业务影响评估和灾难恢复流程。
- 单步骤硬超时限制时间，但调用方仍应使用输出有界的 Skill，避免让自由命令产生大量输出。

## 18. 维护规则

- 新入口必须复用 core，不得绕过 policy、approval、observer 和 audit。
- 新 Skill 必须更新 INDEX、SKILL.md 参数契约、风险声明、脚本和测试。
- 新 MCP transport 或协议版本必须先对照官方规范补连接和失败测试。
- 新策略字段必须同时更新默认配置、校验器、Web 白名单、文档和测试。
- 任何用户可见状态变化必须同步 CLI、机器 API、Web 和审计状态语义。
- 不自动提交代码；发布和版本历史由维护者明确执行。
