项目里“审查”大致分三层：执行前风险审查、策略文件自身校验、执行后审计/观察。它们不是同一种东西。
1. AST/Token 命令守卫
入口在 [lib/command_guard.py (line 1)](/home/free/prassnell/lib/command_guard.py:1)，由 [lib/policy.sh (line 54)](/home/free/prassnell/lib/policy.sh:54) 调用。
它负责看 shell 命令的结构，而不是只匹配字符串。例如：
管道：curl ... | sh
重定向：> /etc/hosts
命令替换：$(...)
包装器：sudo、env、timeout、xargs
交互命令：vim、less、top 未 batch
文件修改：rm、cp、mv、tee、sed -i、curl -o
保护路径：/etc、/boot、/usr、/var/lib、/root、用户 .ssh
它的价值是能识别“命令形态风险”。比如 printf x > /tmp/a 不一定命中正则里的关键字，但 AST 能看到这是文件写入。当前项目要求自由 shell 文件修改走 controlled-tools，所以这类会产生 AST_FILE_MUTATION_REQUIRES_SKILL。
2. 正则风险规则
配置在 [policies/risk-rules.json (line 1)](/home/free/prassnell/policies/risk-rules.json:1)，逻辑在 [lib/policy.sh (line 36)](/home/free/prassnell/lib/policy.sh:36)。
它负责项目可配置的风险规则：
blocked_patterns: 直接阻断，如 rm -rf /、curl | sh
warn_patterns: 高风险但可审批，如 sudo、systemctl restart
remote_script_blocked_patterns: 远程脚本内容专用阻断规则
protected_paths: 保护路径命中规则
protected_services: 关键服务命中规则
正则的优势是易配置、易扩展；缺点是不了解 shell 语法结构。所以它和 AST 是互补关系。
最终聚合在 [lib/policy.sh (line 95)](/home/free/prassnell/lib/policy.sh:95) 的 linux_agent_policy_review_text：
合并 AST findings 和正则 findings
有 critical 就 approved=false
有任何 finding 就 approval_required=true
风险等级按 critical > high > medium > low 汇总
3. Step/执行计划审查
Work plan 的单步审查在 [lib/policy.sh (line 137)](/home/free/prassnell/lib/policy.sh:137)。
它不仅看命令文本，还会考虑模型给出的 risk_level：
模型标记 medium/high/critical，即使命令本身没命中规则，也需要审批
remote_script 会强制提升到至少 high，并要求审批
skill/script/edit 场景会过滤一部分“自由 shell 文件修改”误报，因为 skill 内部是受控能力，不等同于模型直接发 shell
4. 策略文件校验
这是“审查审查规则本身”，在 [lib/policy.sh (line 217)](/home/free/prassnell/lib/policy.sh:217) 之后。
它校验：
JSON 是否合法
必要字段是否存在
正则是否能被 grep -E 或 jq/test 编译
正则是否匹配空字符串，避免全量误判或全量脱敏
audit-boundaries.json 里的 mode 是否在允许列表里
CLI/API/Web 都走这套逻辑，例如：
bash bin/agent policy validate
bash bin/agent api policy validate '{"path":"risk-rules.json"}'
Web 保存策略前也会先校验。
5. Redaction 脱敏规则
配置在 [policies/redaction-rules.json (line 1)](/home/free/prassnell/policies/redaction-rules.json:1)，执行在 [lib/common.sh (line 124)](/home/free/prassnell/lib/common.sh:124)。
它不是阻断规则，而是“写日志/展示前清洗敏感信息”：
API key
Bearer token
Authorization/Cookie
JWT
私钥
长 hex
内网 IP
它用于审计日志、上下文摘要、输出预览，避免把密钥写进日志或 Web 响应。
6. Audit 审计日志
审计主逻辑在 [lib/audit.sh (line 1)](/home/free/prassnell/lib/audit.sh:1)。
Audit 不是执行前拦截器，而是“事实记录层”。它记录：
session 开始/结束
用户请求
sense 结果
AI 调用失败/响应异常
计划生成
step 审查、审批、执行、失败、跳过
terminal/script/edit 结果
observer 结果
它的作用是可追踪、可复盘、可恢复 Web 时间线。Web 里 /api/audit/read 会把审计事件恢复成工作台 timeline，相关逻辑在 [web/server.py (line 968)](/home/free/prassnell/web/server.py:968)。
7. Audit Boundaries 审计边界
配置在 [policies/audit-boundaries.json (line 1)](/home/free/prassnell/policies/audit-boundaries.json:1)。
它定义“审计允许观察什么”：
audit_payload_mode: safe_summary 或 redacted_verbose
audit_text_limit: 文本截断上限
允许记录的 application event
observer 可观察的 syscall
observer 结果字段
observer 最大事件数
它的意义是限制审计系统自己不要变成隐私/敏感信息泄漏源。
8. Observer / auditd 运行时观察
Observer 在 [lib/observer.sh (line 1)](/home/free/prassnell/lib/observer.sh:1)，Web 启用逻辑在 [web/server.py (line 420)](/home/free/prassnell/web/server.py:420)。
当前 lifecycle、`auid` 身份过滤、record id 去重和环境降级规则见 [docs/observer-audit.md](/home/free/prassnell/docs/observer-audit.md:1)。
它和 policy 不同：policy 是执行前预测风险，observer 是执行时/执行后观察实际发生了什么。
它会尝试用 auditd 观察 syscall，例如：
execve
open/openat
rename
unlink
chmod/chown
mkdir/rmdir
如果 auditd 不可用，比如没有权限、WSL、容器环境，它不会阻断主流程，而是记录降级事件。
9. Approval 审批
审批不是独立审查器，而是审查结果的执行闸门。核心在 [lib/executor.sh (line 169)](/home/free/prassnell/lib/executor.sh:169)。
规则大致是：
critical: 阻断
high/medium 或命中 finding: 需要人工审批
low 且策略干净：可按 approvals.auto.* 自动批准
文件 patch、下载、remote script 默认不自动批准
一句话总结
AST：看命令结构，防绕过。
正则：看项目策略，方便配置风险模式。
Policy review：合并 AST + 正则，给出是否阻断/审批/低风险。
Policy validate：校验策略文件本身别写坏。
Redaction：防敏感信息进日志/输出。
Audit：记录事实，用于追踪和回放。
Audit boundaries：限制审计能看什么、记多少。
Observer：运行时观察实际系统行为，补足执行前审查的盲区。
